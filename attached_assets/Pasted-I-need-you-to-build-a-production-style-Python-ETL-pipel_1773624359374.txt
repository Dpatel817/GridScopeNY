I need you to build a production-style Python ETL pipeline for my React/TypeScript NYISO dashboard.

Important architectural requirement:
The ETL must be split into TWO completely separate workflows:

1. A one-time historical backfill script (`backfill.py`)
2. A lightweight nightly incremental updater (`daily_scraper.py`)

Do NOT design this as one giant always-running historical scraper.
The whole point is:
- historical backfill runs once manually
- nightly updater runs every day via GitHub Actions
- no unnecessary reprocessing of old history
- no duplicate rows when daily data overlaps with already loaded data

==================================================
CORE OBJECTIVE
==================================================

Build a Python ETL system that:

1. Backfills historical NYISO data starting at 2024-01-01 and not before that.
2. Uses BOTH:
   - daily file links
   - monthly archived ZIP folders
3. Saves:
   - raw ZIPs and raw CSV/TXT/XLSX files
   - processed CSV outputs
   - deduplicated master Parquet outputs
4. Runs nightly at midnight via GitHub Actions even when the app is not running.
5. Prevents duplicate data when daily files overlap with monthly ZIP archives.
6. Generalizes cleanly across all NYISO endpoints listed below.

==================================================
VERY IMPORTANT ETL STRATEGY
==================================================

This ETL must follow a two-pronged architecture.

----------------------------------
A. ONE-TIME BACKFILL (`backfill.py`)
----------------------------------

This script is run manually exactly one time unless I later decide to backfill even older history.

Requirements:
- Backfill from 2024-01-01 through the current month only
- Use monthly ZIP archives where available
- Iterate month-by-month, not day-by-day if monthly ZIPs exist
- Download ZIP archives
- Save ZIPs to raw storage
- Extract raw files
- Process them
- Save processed CSV
- Save/update Parquet outputs
- Build the foundational historical dataset

After this script runs successfully, it should generally not need to run again.

It should NOT be part of the nightly job.

----------------------------------
B. NIGHTLY INCREMENTAL UPDATER (`daily_scraper.py`)
----------------------------------

This script is the one triggered nightly via GitHub Actions.

Requirements:
- It should NOT run the full historical backfill
- It should only pull recent daily files
- It should fetch a rolling 7-day window for dated endpoints
- This rolling window is intentional as a recovery mechanism for missed cron runs, NYISO outages, or delayed files
- It should process only recent data
- It should append/upsert to the existing Parquet outputs
- It should deduplicate before writing so repeated 7-day overlap never creates duplicate rows

Example:
If the job fails on Tuesday, Wednesday’s run should still capture Tuesday’s data because Tuesday is inside the rolling 7-day lookback window.

This script must remain lightweight and resilient.

==================================================
IDEMPOTENCY / DEDUPLICATION REQUIREMENTS
==================================================

This is critical.

The same data may arrive from:
- monthly ZIP archives during the initial backfill
- daily CSV endpoints during nightly updates
- overlapping rolling 7-day fetches across nightly runs

The system must never create duplicates.

Requirements:
1. Maintain manifest files in manifests/ tracking:
   - downloaded ZIPs
   - extracted raw files
   - processed daily dates
   - processed monthly archives
   - last successful run time
2. Before writing to a master Parquet file:
   - load existing Parquet if present
   - concatenate with new data
   - drop duplicates using dataset-specific primary keys
   - keep last version
3. Save Parquet partitioned if helpful, but keep querying simple
4. Optionally hash raw files so extraction/processing is not repeated unnecessarily

General dedupe rule:
- use dataset-specific primary keys when possible
- otherwise use a normalized full-row hash fallback
